## Critical Issues (App-breaking)

### ISS-001 тАФ Subtask reorder route never reached
- **Status**: ЁЯЯв Fixed
- **File**: `routes/api.php` (lines 72-74)
- **Description**: The `{subtask}` parameter route is defined before the `/reorder` named route. Laravel matches `reorder` as a subtask ID, so `POST /tasks/{task}/subtasks/reorder` is never reached.
- **Fix**: Move the `reorder` route BEFORE the `{subtask}` parameter route.

---

### ISS-002 тАФ reorderTasks uses PATCH instead of POST
- **Status**: ЁЯЯв Fixed
- **File**: `resources/js/stores/tasks.js` (line 309)
- **Description**: The `reorderTasks` action uses `axios.patch()` but the backend route `POST tasks/{task}/move` expects a POST request. Results in 405 Method Not Allowed.
- **Fix**: Change `axios.patch` to `axios.post`.

---

### ISS-003 тАФ Labels never synced after task create/update
- **Status**: ЁЯЯв Fixed
- **File**: `resources/js/components/kanban/TaskModal.vue` (lines 180-197)
- **Description**: TaskModal includes labels in `formData` but the backend `StoreTaskRequest` does not accept a `labels` field. The separate `syncLabels` endpoint (`POST /tasks/{task}/labels`) is never called after task creation or update. Labels appear selected in the UI but are never saved.
- **Fix**: After successful `createTask`/`updateTask`, call `tasksStore.syncLabels(taskId, formData.labels)` if labels were selected.

---

### ISS-004 тАФ Assignee filter uses plural `assignees` instead of singular `assignee`
- **Status**: ЁЯЯв Fixed
- **File**: `resources/js/composables/useTaskFiltering.js` (lines 43, 139)
- **Description**: The filtering logic references `task.assignees` (plural array) but the Task model has a `BelongsTo` relationship returning `task.assignee` (singular object). Assignee filtering always returns empty results.
- **Fix**: Change `task.assignees` to `task.assignee` and adjust array logic to handle single object.

---

### ISS-005 тАФ CommentController logs wrong activity subject_type
- **Status**: ЁЯЯв Fixed
- **File**: `app/Http/Controllers/CommentController.php` (line 109)
- **Description**: Activity logging uses `Comment::class` as `subject_type` and `comment_id` as `subject_id`. All task-related activities should use `Task::class` and the task's ID so the activity feed displays correctly.
- **Fix**: Change `subject_type` to `Task::class` and `subject_id` to `$task->id`.

---

### ISS-006 тАФ ProjectтЖТtasks hasManyThrough relationship broken
- **Status**: ЁЯЯв Fixed
- **File**: `app/Models/Project.php` (lines 108-119)
- **Description**: The `hasManyThrough` relationship tries to go ProjectтЖТColumnтЖТTask but skips the Board intermediary. Uses a manual `join('boards', ...)` hack that may return tasks from other projects.
- **Fix**: Rewrite relationship to properly chain through BoardтЖТColumnтЖТTask, or use a query scope with proper joins.

---

## High Issues (Broken features)

### ISS-007 тАФ Due date validation rejects today's date
- **Status**: ЁЯЯв Fixed
- **File**: `app/Http/Requests/StoreTaskRequest.php` (line 30), `app/Http/Requests/UpdateTaskRequest.php` (line 30)
- **Description**: Validation rule `after:today` prevents setting due dates for today. Users get unexpected 422 errors when creating tasks due today.
- **Fix**: Change `after:today` to `after_or_equal:today`.

---

### ISS-008 тАФ projectMembers not passed to FilterBar
- **Status**: ЁЯЯв Fixed
- **Files**: `resources/js/components/kanban/KanbanBoard.vue` (lines 5-10), `resources/js/components/kanban/BoardHeader.vue` (lines 5-10)
- **Description**: KanbanBoard does not pass `projectMembers` to BoardHeader, which in turn cannot forward it to FilterBar. The assignee filter dropdown renders empty.
- **Fix**: Fetch project members in KanbanBoard and pass down through BoardHeader to FilterBar.

---

### ISS-009 тАФ Submit error not displayed in TaskModal template
- **Status**: ЁЯЯв Fixed
- **File**: `resources/js/components/kanban/TaskModal.vue` (template section)
- **Description**: When API calls fail, the error is stored in `errors.value.submit` but there is no corresponding `<span>` or error element in the template to display it. Users see no feedback on failure.
- **Fix**: Add error message rendering for `errors.submit` in the template.

---

### ISS-010 тАФ Missing project authorization in TaskController@index
- **Status**: ЁЯЯв Fixed
- **File**: `app/Http/Controllers/TaskController.php` (lines 24-35)
- **Description**: The `index()` method does not verify that the authenticated user has access to the requested `project_id`. A user could fetch tasks from projects they don't belong to.
- **Fix**: Add authorization check to verify user is a member of the project.

---

### ISS-011 тАФ CommentController@destroy missing task context check
- **Status**: ЁЯЯв Fixed
- **File**: `app/Http/Controllers/CommentController.php` (lines 82-92)
- **Description**: The `destroy()` method does not verify that the comment belongs to the given task. A user could delete comments from other tasks if they know the comment ID.
- **Fix**: Add check that `$comment->task_id === $task->id` before deletion.

---

### ISS-012 тАФ N+1 query fallback in TaskResource
- **Status**: ЁЯЯв Fixed
- **Files**: `app/Http/Resources/TaskResource.php` (lines 27-30), `app/Http/Resources/TaskDetailResource.php` (lines 27-30)
- **Description**: The `->count()` fallbacks (`$this->subtasks()->count()`) trigger individual queries per task when `loadCount()` is not applied by the controller. Causes performance degradation on boards with many tasks.
- **Fix**: Ensure all controllers use `loadCount('subtasks', 'comments', 'labels')` before returning TaskResource collections, or default to 0 instead of querying.

## Medium Issues (UX/Quality)

### ISS-013 тАФ Task position uses 0-based index
- **Status**: ЁЯЯв Fixed
- **File**: `app/Http/Controllers/TaskController.php` (line 307)
- **Description**: `reorderTasksInColumn()` uses 0-based `$index` as position value. If the rest of the system expects 1-based positions, ordering will be inconsistent.
- **Fix**: Use `$index + 1` or verify all consumers expect 0-based positions.

---

### ISS-014 тАФ TaskDetailModal uses `task.status` which may be undefined
- **Status**: ЁЯЯв Fixed
- **File**: `resources/js/components/kanban/TaskDetailModal.vue` (line 29)
- **Description**: Template references `task.status` but the Task model uses `column_id` to determine status. The field may render as undefined.
- **Fix**: Derive status from the task's column name or use a computed property.

---

### ISS-015 тАФ Missing column_id validation in TaskModal
- **Status**: ЁЯЯв Fixed
- **File**: `resources/js/components/kanban/TaskModal.vue` (line 180)
- **Description**: `validateForm()` only validates title length but does not check that `column_id` is set. Submitting without a column causes a 422 API error instead of a friendly UI message.
- **Fix**: Add `column_id` check to `validateForm()`.

---

### ISS-016 тАФ Missing Str facade import in CommentController
- **Status**: ЁЯЯв Fixed
- **File**: `app/Http/Controllers/CommentController.php` (line 46)
- **Description**: Uses `\Str::limit()` without importing `Illuminate\Support\Str`. Works with the `\Str` global alias but is fragile and inconsistent with project conventions.
- **Fix**: Add `use Illuminate\Support\Str;` to the imports.
